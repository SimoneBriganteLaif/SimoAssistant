La diagnosi

Gli 8 problemi → la stessa causa radice

Tutta la flotta nasce dallo stesso TemplateStack (804 righe, 56 parametri). 10 repo su 24 hanno app.py byte-identico. I problemi del brief sono scritti nei default della libreria.

#1 · #3

Zero monitoring white-box

Nessun alarm in 3.970 righe di construct. Container Insights / Performance Insights off. I down li scoprono i clienti.

#2 · #7

Niente scalabilità

L'ALB punta all'ASG di istanze, non al servizio ECS. Bridge + hostPort fisso = max 1 task/istanza. max_capacity=1.

#6

Database pubblici

VPC con sole subnet pubbliche → RDS PubliclyAccessible=true con IP pubblico. Verificato live su Nivi e Casartelli.

#4

ETL/task caotici

5 varianti copia-incollate: ETL su EC2 h24, background task, workflow gemello FLG_ETL, ~1.640 righe di Step Functions custom.

#5

laif-cdk sporco

4 stack su 14 usati; 8 mai importati. Zero test reali. 705 righe di AWS imperativo fuori da CloudFormation.

#8

Account isolati

~60 account/VPC separati. Isolamento prezioso ma costoso: il floor fisso è metà bolletta. Scelta: tenere l'isolamento.

Tooling

Deploy dal laptop

0/24 repo con CI. Container Docker da 764 MB emulato su Mac ARM. Access key statiche larghissime. ~21-37 passi per onboarding.

Monitoring reale

Wolico (non Checkmk)

Checkmk è un rimasuglio mai usato. Il monitoring passa da Wolico (errori app) — da arricchire col white-box infra.

Caso di studio — la prova

"Nivi va giù": lo scale-up a mano era la risposta sbagliata

Cosa è successo

Un health check troppo aggressivo (curl / ogni 5s, timeout 5s, 2 retry) uccide l'unico task ECS appena il backend single-worker satura 1 vCPU (CPU inchiodata al 50% di un host 2-vCPU). Con desired_count=1 ogni kill = 2-5 min di 502/503.

127 min di down (27/5–10/6) upsize: +26,6 $/mese (+103%) non ha risolto

L'upsize EC2 t4g.small→medium ha gli stessi 2 vCPU: il collo di bottiglia era 1 worker, non la RAM. Il 10/6, dopo l'upsize, il task è stato ucciso ancora 3 volte.

Il fix vero costa 0 $

Defangare l'health check (30s, /health)0 $
2+ worker (sfrutta il 2° vCPU già pagato)0 $
desired_count=2 dietro l'ALB già pagato0 $
Circuit breaker + rollback automatico0 $
4 alarm CloudWatch → Wolico~0 $

Questo pattern è nei default di tutta la flotta. Si corregge una volta nella libreria → vale per tutti.

Infra nella repo di progetto

Una scheda laif.yaml (~15 righe) sostituisce la repo <cliente>-infra. N-app/1-account via apps:[].

La scheda values.yaml — tutto qui dentro

# prima-power: 2 app sullo stesso account, senza forkare app.py
identity: { customer: prima-power, accounts: { dev:…, prod:… }, region: eu-west-1 }
defaults:            # sicuri-by-default
  monitoring: true     # CloudWatch → Wolico
  db_private: true     # RDS PRIVATE_ISOLATED
  waf: true            # era deployato 0/24
  multi_az: false      # opt-in solo prod
  autoscaling: false   # opt-in per-app
apps:
  - name: prima-power   # naming derivato: dominio, stack, bucket, secret
    db: { name: prima_power, multi_az: true }
    flags: { autoscaling: true }
    jobs:               # il modulo JOBS si compone solo se questa lista esiste
      - { name: inventory, kind: scheduled, size: medium, schedule: "cron(30 5 * * ? *)" }
  - name: prima-pilot   # 2ª app, stesso account
    flags: { auto_turnoff: true, waf: false }

I 5 rischi da sorvegliare

1. Cutover di rete = REPLACE (blue/green sul CNAME).

2. Interface endpoint vs API esterne (mappa per cliente).

3. Cert cross-account (delega di zona prima).

4. Config drift repo↔prod (cdk diff è l'autorità).

5. "Stabilizzare due volte" il tooling (convergere sul nuovo CDK).
